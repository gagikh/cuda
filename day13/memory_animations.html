Each "request" is a load issued by a warp. Most resolve close to the SM (fast); a minority miss all the way to global memory, which is ~15× slower than L1 — this is the same pyramid as cache_hierarchy.svg, but now you can see where traffic actually lands. `__ldg()`, L2 persistence hints, and shared-memory swizzling (this day) are all about biasing more requests toward the top of this pyramid.

Coalesced

Strided

thread i reads addr = base + i×32 · each request lands in its own line
0
Same 32 threads, same 128 bytes of useful data, two access patterns. Coalesced access packs all 32 requests into one transaction because they're contiguous. Strided access scatters them across 32 separate cache lines, so the memory controller has to issue 32 transactions to fetch the same 128 bytes of useful data — the rest of each fetched line is wasted bandwidth. `__ldg()` and read-only caching help regardless of pattern, but they don't fix a bad access pattern; only the pattern itself does.
