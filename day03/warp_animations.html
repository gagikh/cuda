6 warps resident on one partition. After a warp issues, it goes "not ready" for a few cycles (standing in for a memory or execution latency) before it can issue again. Because several warps are resident at once, the scheduler almost always has someone ready — that's latency hiding, the reason GPUs lean on massive multithreading instead of branch prediction and deep caches. Watch the idle counter: it should barely move. This is the same fetch → decode → scheduler → lockstep issue pipeline from pipeline.svg, run forward across many cycles.

if (tid < 16) A(); else B();

A warp issues one instruction to all 32 lanes at once. When threads disagree on a branch, the warp can't split — it runs path A with the other lanes masked off, then runs path B with the first set masked off. Cost: this divergent warp takes ~5 cycles instead of the ~3 it'd take if every thread agreed on the branch. Divergence doesn't add parallelism, it removes it, one path at a time.
